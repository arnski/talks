JavaScript Eckpunkte (90ies)

  • 1995: Brendan Eich entwickelt JS - in 10 Tagen
  • 1996: Netscape Navigator 3 - JavaScript 1.1
  • 1996: Internet Explorer 3 - JScript <= Beginn des Browserkrieges
  • 1996: Mit Netscape "Livewire" erster Versuch JS => Server
  • 1999: Huch! MS erfindet XMLHTTPObject. AJAX, Baby
Zu Brendan entwickelt JS - in 10 Tagen: Laut eigener Aussage (Engine). N.B. Brendan Eich ist Mitbegründer und aktueller CTO von Mozilla. Zu IE3 / JScript: Wir erinnern uns: Microsoft hat hier eine eigene Implementation hingelegt, die natürlich nicht komplett deckungsgleich mit JavaScript war.

JavaScript Eckpunkte (00er)

  • 2000: Douglas "The Crock" Crockford entdeckt *JSON* als Objektaustausch
  • 2004: Gmail als SPA
  • 2005: jQuery
  • 2005: Adobe CS2 - skriptbar via JS
  • 2006: CouchDB bringt JS in die DB- Welt
  • 2008: Google Chrome & V8 - ungeahnte Performance => Engine Wars
  • 2009: Ryan Dahl entwickelt Node.js
Zu jQuery: Hat es den bis dato verwirrten Frontend- Devs möglich gemacht, mit dem DOM umzugehen. Und damit Javascript attraktiv. Javascript ist u.a. durch jQuery extrem populär geworden. Zu V8: Google kommt mir v8 heraus. Sie sehen es als einen Weg, den das Web / den Browsermarkt (denken wir auch an Gmail) zu dominieren. Auf einmal stehen die anderen JS- Engine Hersteller unter Zugzwang auch zu optimieren. Und schwups - quasi über Nacht - ist JS schneller als Ruby und Python. V8 steigerte die Leistung, indem u.a. der JavaScript-Code bei der Ausführung zunächst in nativen Maschinencode übersetzt wird (Just-in-time-Kompilierung), anstatt Bytecode auszuführen oder die Skriptsprache direkt zu interpretieren.

Rewind: 2005-2008

  • Rails & Fullstackframeworks
  • Performance- Schwierigkeiten
  • Lösungsansatz: Merb (sf2?!)
  • * "No Code is faster than no code" :D
  • Ryan Dahl - Ruby is at fault
Spulen wir nochmal ein wenig zurück: Erinnern wir uns an die Jahre 2005-2008. Rails explodierte. Und mit Rails auch die Full- Stack Frameworks der anderen Scriptsprachen. Ein viel diskutiertes Problem dieser Zeit war die Performance der Frameworks, insbesondere Rails. Skalierbarkeit != Performance Es gab einige Versuche, Rails schneller zu machen - u.A. Merb (Ein bisschen wie sf2 - Coole Tagline "No Code is faster than no code"). Merb hat quasi eine Reinraumversion des Rails Controllers genommen, und alles weitere in Module ausgelagert, die bei Bedarf hinzugenommen wurden. Merb behob unter anderem das "Request-Lock" von Rails: Bis die Response raus war, nichts anderes machen. Nach all den Mühen ist man jedoch schliesslich an den Punkt gekommen, an dem man gemerkt hat, dass das Problem an ganz anderer Stelle lag. Nicht Rails - sondern Ruby selbst hat das Performanceproblem. Es hatte, ähnlich wie Python ein eher beschränktes Threading Modell, Interpreter Locks, etc.. Ich möchte mich hier nicht im Detail verlieren, unterm Strich steht jedoch, dass damals die Parallelisierung von Aufgaben wie z.B. Requestabarbeitung sehr schlecht funktionierte. Er experimentierte nun zunächst mit ... So. Das war jetzt eine ganze Menge Kontext. Doch ...
